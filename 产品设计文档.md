# ipcheck 产品设计文档

## 1. 项目背景

当前 Xray WebSocket 真实客户端 IP 统计通过脚本推送到 Uptime Kuma Push Monitor。Uptime Kuma 适合判断 Push 是否按时到达、服务是否 Up，但它的事件区只展示状态变化，不适合实时查看每次 Push 的详细统计内容。

ipcheck 的目标是补足这一点：把 ws/ws1/ws2/ws3 的连接日志解析成可视化监控面板，用一个页面展示最近连接趋势、入口分布、TOP IP 和最新统计结果。

## 2. 项目目标

- 展示最近更新时间和当前数据状态。
- 展示最近 7 天总连接数。
- 展示最近 7 天独立 IP 数。
- 展示 TOP 10 IP，包含 IP、连接次数、来源入口 ws/ws1/ws2/ws3。
- 支持 30 分钟、24 小时、7 天、30 天、90 天统计窗口。
- 保存 180 天历史快照，用于趋势图和历史对比。
- 使用 ECharts 做更直观的趋势、分布、排行和热力展示。
- 部署到当前服务器，通过 `ipcheck.shandawang.cc` 公网访问。
- 第一版不做登录，不做设置页。

## 3. 非目标

- 不替代 Uptime Kuma 的告警能力。
- 不修改现有 Uptime Kuma Push 脚本。
- 不做复杂用户系统、角色权限或后台管理。
- 不接入第三方 IP 地理位置接口，避免额外依赖和隐私风险。
- 不保存原始访问日志全文，只保存面板需要的聚合快照和必要明细。

## 4. 用户与访问方式

唯一用户是服务器维护者本人。页面会通过 Cloudflare 橙云域名 `ipcheck.shandawang.cc` 公开访问。

第一版应用代码不内置登录系统，但公网访问前必须通过 1Panel/OpenResty 反代密码保护。由于页面会完整显示真实 IP，不能只依赖域名不公开传播。后续如需更强保护，可升级为 Cloudflare Access。

## 5. 数据来源

数据来自当前 1Panel/OpenResty 站点访问日志：

- `ws.shandawang.cc`
- `ws1.shandawang.cc`
- `ws2.shandawang.cc`
- `ws3.shandawang.cc`

只统计成功 WebSocket 连接，主要过滤条件：HTTP 状态码 `101`。

真实客户端 IP 使用日志末尾的 `X-Forwarded-For` 字段，而不是第一列 Cloudflare 边缘 IP。

## 6. 数据窗口

页面支持以下统计窗口：

- 最近 30 分钟
- 最近 24 小时
- 最近 7 天
- 最近 30 天
- 最近 90 天

数据保留 180 天。超过 180 天的历史快照可以由清理任务自动删除。

## 7. 页面信息架构

ipcheck 第一版采用单页 Dashboard。

### 7.1 顶部 Hero 区

内容：

- 产品名：`ipcheck`
- 副标题：`Xray WebSocket Real IP Dashboard`
- 当前状态：正常 / 异常 / 数据延迟
- 最近更新时间
- 当前统计窗口切换按钮
- 下次自动刷新倒计时

### 7.2 核心指标卡片

展示：

- 最近 7 天总连接数
- 最近 7 天独立 IP 数
- 当前窗口总连接数
- 当前窗口独立 IP 数
- 最活跃入口
- TOP IP 连接数

### 7.3 图表区

图表：

- 总连接趋势折线图：展示连接数随时间变化。
- 分入口趋势折线图：ws/ws1/ws2/ws3 各自趋势。
- 入口分布环形图：四个入口占比。
- 入口对比柱状图：四个入口连接数对比。
- TOP 10 IP 横向排行榜：IP + 次数 + 入口构成。
- 时间热力图：展示一天中各时间段连接活跃度。
- Push/采集状态时间线：展示最近采集任务是否成功。

### 7.4 最新统计详情

展示最新一次采集结果：

- 采集时间
- 统计窗口
- 总连接数
- 独立 IP 数
- TOP 10 IP
- ws/ws1/ws2/ws3 分入口统计
- 原始摘要文本，格式接近当前 Uptime Kuma Push 文案

### 7.5 最近历史

展示最近 20 次采集快照：

- 时间
- 状态
- 总连接数
- 独立 IP 数
- TOP IP
- 四入口计数

## 8. 视觉方向

用户希望页面更有设计感，不要普通后台模板。建议采用“深紫黑 + 荧光绿 + 数据玻璃卡片”的监控视觉语言，参考 Sentry 风格但不照搬。

### 8.1 视觉关键词

- 深色技术感，但避免纯黑。
- 卡片有玻璃质感和层次。
- 数字大而清晰，适合一眼扫读。
- 图表颜色有明显区分，但整体统一。
- 使用动效表达数据刷新，不做花哨无意义动画。

### 8.2 色彩建议

- 背景：`#1f1633`、`#150f23`
- 卡片：`rgba(255,255,255,0.08)`
- 边框：`rgba(194,239,78,0.18)` 或 `#362d59`
- 主文本：`#ffffff`
- 次级文本：`#c9c2d8`
- 重点色：`#c2ef4e`
- 辅助色：`#6a5fc1`、`#fa7faa`、`#ffb287`

### 8.3 字体建议

- UI 字体：Rubik
- 数字 / 技术标签：JetBrains Mono

### 8.4 响应式

- 桌面端：多列 Dashboard，图表并排展示。
- 平板：两列布局。
- 手机：单列卡片，图表高度降低，TOP IP 表格改成卡片列表。

## 9. 技术架构

### 9.1 推荐技术栈

- 前端：Vue 3 + Vite + TypeScript
- 图表：ECharts + vue-echarts
- 状态管理：Pinia 或轻量 composables
- 后端：Node.js + Fastify + TypeScript
- 数据库：SQLite
- ORM/SQL：第一版可用 better-sqlite3，避免复杂 ORM
- 调度：systemd timer 每 10 分钟运行采集脚本
- 部署：Node 服务托管 API 和前端静态文件，OpenResty/1Panel 反代到本地端口

### 9.2 服务边界

建议项目包含三个运行部分：

1. 采集脚本
   - 每 10 分钟执行一次。
   - 读取 OpenResty 访问日志。
   - 解析状态码 101 的 WebSocket 连接。
   - 写入 SQLite 快照表和聚合表。

2. API 服务
   - 提供 Dashboard 所需 JSON 数据。
   - 只监听 `127.0.0.1`。
   - 由 OpenResty 反代到 `ipcheck.shandawang.cc`。

3. 前端页面
   - Vite build 生成静态文件。
   - 由 API 服务统一托管，减少部署复杂度。

## 10. 数据模型设计

### 10.1 snapshots

保存每次采集快照。

字段：

- `id`
- `created_at`
- `window_minutes`
- `total_connections`
- `unique_ips`
- `status`
- `message`
- `raw_summary_json`

### 10.2 snapshot_entry_counts

保存每次快照中每个入口的连接数。

字段：

- `id`
- `snapshot_id`
- `entry_name`，取值 `ws`、`ws1`、`ws2`、`ws3`
- `connection_count`

### 10.3 snapshot_top_ips

保存每次快照的 TOP IP。

字段：

- `id`
- `snapshot_id`
- `ip`
- `connection_count`
- `entry_breakdown_json`
- `rank`

### 10.4 collection_runs

保存采集任务状态。

字段：

- `id`
- `started_at`
- `finished_at`
- `status`
- `logs_read`
- `lines_scanned`
- `connections_matched`
- `error_message`

## 11. API 设计

### 11.1 `GET /api/health`

返回服务健康状态。

### 11.2 `GET /api/summary`

返回首页核心摘要。

查询参数：

- `range=30m|24h|7d|30d|90d`

返回：

- 最近更新时间
- 当前窗口总连接数
- 当前窗口独立 IP 数
- 最近 7 天总连接数
- 最近 7 天独立 IP 数
- 入口统计
- TOP 10 IP
- 当前状态

### 11.3 `GET /api/timeseries`

返回趋势图数据。

查询参数：

- `range=30m|24h|7d|30d|90d`
- `bucket=10m|1h|1d`

### 11.4 `GET /api/top-ips`

返回 TOP IP 排行。

查询参数：

- `range=30m|24h|7d|30d|90d`
- `limit=10|20|50`

### 11.5 `GET /api/snapshots/recent`

返回最近 N 次采集快照。

查询参数：

- `limit=20`

### 11.6 `GET /api/entries`

返回 ws/ws1/ws2/ws3 分入口统计。

## 12. 自动刷新策略

前端每 10 分钟自动刷新一次数据，并显示倒计时。用户也可以手动刷新。

第一版不使用 WebSocket/SSE，避免增加复杂度。后续如果需要更实时，可以增加 SSE。

## 13. 安全与隐私

- 页面完整显示 IP，仅供个人使用。
- API 服务只监听 `127.0.0.1`，不直接暴露端口。
- 通过 Cloudflare + OpenResty 访问。
- SQLite 数据库权限限制为服务用户可读写。
- 不在 GitHub 提交数据库文件、日志文件、`.env`、部署密钥。
- 若后续需要更稳妥访问控制，优先加 Cloudflare Access 或 1Panel Basic Auth。

## 16. 设计风险与建议

### 16.1 不登录但公网访问

当前决定是不需要登录，但页面会完整显示 IP。这个设计能最快上线，但隐私风险较高。建议第一版至少保留一键开启 Basic Auth 或 Cloudflare Access 的部署说明。

### 16.2 10 分钟统计与日志量

如果每次都扫描大量历史日志，性能会逐渐变差。第一版可以先通过 `tail_lines` 控制扫描范围；正式实现应保存采集游标或按时间窗口高效读取。

### 16.3 历史趋势的准确性

如果只保存每 10 分钟快照，趋势图代表“每次统计时刻的窗口值”，不是逐条连接事件的精确时序。若后续要更准确的趋势，应保存事件级聚合数据，例如每 10 分钟、每入口、每 IP 的增量统计。

### 16.4 日志轮转

OpenResty 日志可能会轮转。采集脚本需要兼容当前日志和 rotated 日志，否则 30/90 天统计可能不完整。第一版应在页面显示“数据覆盖范围”提示。

## 17. 推荐第一版 MVP

第一版优先实现：

- 新采集脚本，不修改现有 Uptime Kuma 推送脚本。
- SQLite 数据库。
- Fastify API。
- Vue + ECharts 单页 Dashboard。
- 最近更新时间、7 天总连接数、7 天独立 IP 数、TOP 10 IP。
- 30m/24h/7d/30d/90d 窗口切换。
- 每 10 分钟自动刷新。
- systemd timer 每 10 分钟采集。
- 反代到 `ipcheck.shandawang.cc`。
